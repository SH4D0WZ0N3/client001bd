01

What This Bot Does

📋

Content Copying

Reads messages from a private source channel and copies them — without any "Forwarded from" label — to your public channel or group.

🖼️

Album Support

Handles multi-photo and multi-video albums correctly. All images in a group are sent together as a single album, preserving the original structure.

✏️

Custom Captions

Appends your watermark or channel link to every post caption automatically. Original captions are preserved; your text is added below them.

⏱️

Scheduled Posting

Posts one item from the queue at a configurable interval (e.g. every 30 minutes). You control exactly how fast content is published.

📊

Daily Limits

Caps the total number of posts per day. At midnight, the counter automatically resets and posting resumes the next day.

🔄

Queue Persistence

All pending content is stored in MongoDB. If the bot restarts for any reason, the queue is fully recovered and posting continues exactly where it left off.

02

System Workflow

The bot runs three independent processes simultaneously: a message listener, a queue manager, and a scheduler.

Process 1 — Listener

Pyrogram client watches the source channel 24/7. Every new message triggers the handler immediately.

Process 2 — Queue Manager

Buffers album messages, coalesces them, and inserts all content into the MongoDB queue in the correct order.

Process 3 — Scheduler

Every N seconds, dequeues the oldest pending item and sends it to the target channel. Enforces daily limit.

How Albums Work

⚠️
The Album Problem

When someone sends an album (multiple photos/videos at once) on Telegram, each image arrives as a separate individual message to the bot. They all share the same media_group_id, but they arrive as separate events, often milliseconds apart. Without special handling, each photo would be sent individually, breaking the album.

The Solution — 3-Second Coalescing

Added to in-memory buffer. 3-second timer starts.
t=0s
2
Photo 2 arrives (same album)
Added to buffer. Timer RESETS to 3 seconds.
t=0.1s
3
Photo 3 arrives (same album)
Added to buffer. Timer RESETS again.
t=0.2s
4
No more messages for 3 seconds
Timer fires. Buffer is locked, sorted by message ID, flushed to MongoDB as ONE queue item.
t=3.2s
5
Scheduler sends the album
copy_media_group sends all photos in one API call. Album arrives intact at the target channel.
sent
✓ Crash Safety

If the bot crashes mid-album, the unique index on message_id prevents duplicates when it restarts. The album is re-buffered cleanly.

✓ Order Guarantee

All messages are sorted by their numeric ID before sending. Photos always appear in the same order as the original album.

The counter resets at the first scheduler tick after midnight. If TIMEZONE=America/New_York, the reset happens at midnight New York time, not UTC.

Atomic Dequeue — No Double Sends

When the scheduler dequeues an item, it uses MongoDB's find_one_and_update — a single atomic operation that both finds the item AND changes its status to "processing" simultaneously.

// What happens in one atomic MongoDB operation:
{
  "filter": { "status": "pending" },
  "update": { "$set": { "status": "processing" } },
  "sort":   [["message_id", 1]]
}

Even if the scheduler fires twice at the exact same millisecond, they cannot claim the same item.

Unique Index — No Duplicate Inserts

MongoDB enforces a unique index on message_id in the queue collection. If the bot tries to insert the same message twice (e.g., after a crash + restart), MongoDB rejects the duplicate silently.

// MongoDB index (created automatically at startup)
db.queue.createIndex(
  { "message_id": 1 },
  { unique: true }
)

This is the final safety net. Even if the in-memory deduplication is bypassed, the DB prevents duplicates at the storage level.

08

Restart Safety

The bot is designed to survive any restart — container crash, redeployment, server reboot — without losing a single message.

🍃

MongoDB Queue

Every pending item exists in MongoDB before the bot sends it. Restart = resume from same position. Queue never resets.

📍

State Cursor

The last_processed_message_id in MongoDB tracks which messages have been queued. On restart, the bot knows exactly where it left off.

🔐

Session File

Pyrogram's authentication session is stored in a persistent volume. The bot re-authenticates automatically on first run; subsequent restarts reuse the existing session.

What Happens After a Crash

Bot crashes or is restarted
In-memory album buffers are lost. Any items that were in "processing" status are stuck.
Bot restarts and connects to MongoDB
No manual intervention needed. MongoDB connection is automatic.
Initial scan check
Bot reads last_processed_message_id from state. If already scanned, skips bootstrap. If new messages arrived during downtime, they are detected and queued.
Scheduler starts
APScheduler job state is also stored in MongoDB. It resumes its interval immediately.
Posting resumes
Any items stuck in "processing" should be manually reset to "pending" if needed (see troubleshooting guide).

FloodWait Protection

🌊
What is FloodWait?

Telegram rate-limits all bots. If the bot sends too many API requests too quickly, Telegram responds with a FloodWait error telling the bot exactly how many seconds to wait before trying again.

How the Bot Handles It
FloodWait error received
Telegram says "wait X seconds"
Item re-queued as "pending"
The current item is NOT marked failed. It goes back to the front of the queue.
Scheduler interval provides natural backoff
The next tick won't fire until SEND_INTERVAL_SECONDS passes, naturally respecting the rate limit.
Item is retried on next tick
No messages are skipped or lost due to FloodWait.
Prevention Strategy

Setting a generous SEND_INTERVAL_SECONDS greatly reduces FloodWait risk:

SEND_INTERVAL_SECONDS=60 High risk
SEND_INTERVAL_SECONDS=600 Moderate
SEND_INTERVAL_SECONDS=1800 Recommended
Zero Data Loss Guarantee

FloodWait never causes a message to be skipped. The item stays in the queue and is retried on the next scheduler tick. The only effect is a slight delay in posting.

Step-by-Step Posting Lifecycle

01
Admin uploads content to source channel

Photo, video, album, or text post is sent to the private source channel by an admin. The channel is accessible only to the bot.

02
Pyrogram receives the message event

The message handler fires instantly. For albums, each photo arrives as a separate event. For single messages, one event fires.

03
QueueManager buffers or inserts directly

Single messages → immediate DB insert. Album messages → 3-second buffer until all parts arrive, then single DB insert for the group.

04
APScheduler tick fires after interval

Every SEND_INTERVAL_SECONDS, the scheduler fires the posting_job. Only one instance can run at a time.

05
Daily limit check

If today's date changed since last run, reset the counter. If daily_sent_count ≥ DAILY_LIMIT, exit — no post today.

06
Oldest pending item dequeued atomically

MongoDB find_one_and_update finds the oldest "pending" item and marks it "processing" in one atomic operation.

07
Caption built and applied

Original caption (if any) is preserved. FIXED_CAPTION is appended below a separator. HTML formatting is maintained.

08
Telegram API call made

copy_message for single items, copy_media_group for albums. No "Forwarded from" label. Content appears as original.

09
Status updated in MongoDB

Item marked "sent". A record written to sent_logs. daily_sent_count incremented atomically.

10
Subscribers see the post

The message appears in the target channel with your custom caption. Full quality — no compression, no forwarding label.

FAQ

Does the bot add "Forwarded from" to posts?

No. The bot uses Telegram's copy_message API which copies the content without any forwarding label. Posts appear as if the target channel posted them directly.

What happens if the bot goes offline for several hours?

Nothing is lost. The queue in MongoDB stores all pending content. When the bot comes back online, it resumes posting exactly where it left off. Messages sent to the source channel during the downtime are detected on restart via the initial scan catch-up logic.

Can I post to multiple target channels?

The current setup supports one target channel (TARGET_CHAT_ID). To post to multiple channels, you would need to run multiple bot instances with different configurations.

Will it resend old content every time it restarts?

No. The last_processed_message_id in MongoDB tracks which messages have already been scanned and queued. On restart, the initial scan is skipped if this value is greater than START_MESSAGE_ID. Already-sent items have status "sent" and are never re-sent.

What if Telegram rate-limits the bot?

The bot handles FloodWait automatically. The affected item is re-queued and retried on the next scheduler tick. No manual action is needed. To reduce the risk of FloodWait, increase SEND_INTERVAL_SECONDS.

Can I change DAILY_LIMIT or SEND_INTERVAL without restarting?

These are loaded from environment variables at startup. To change them, update the variable in Railway's dashboard and trigger a redeploy. The new values take effect immediately after restart.

What file types are supported?

Everything Telegram supports: photos, videos, documents, audio, voice messages, and text posts. Albums (multiple photos/videos sent together) are handled as single grouped sends.

Is the source channel content modified before posting?

Only the caption is modified. Original media files are never re-uploaded or re-compressed — Telegram's copy API copies the file reference at the API level, which means instant delivery at full original quality.

How do I know if the bot is working?

Check Railway logs for entries like INFO: Successfully sent item from source message X. You can also check MongoDB: db.queue.countDocuments({status:"sent"}) should increase over time.

What's the minimum Railway plan needed?

The free tier has limited usage hours per month — not enough for 24/7 operation. For a continuously running bot, you need the Hobby Plan ($5/month). This gives unlimited uptime and is sufficient for this bot.
